Skills in OpenAI API

本文介绍如何通过 OpenAI API 上传、管理和附加可复用的 Skills 到托管环境。Skills 是以 SKILL.md 为锚定的文件夹包，模型可按需发现并执行。本文涵盖 Skills vs. 工具 vs. 系统提示的区别、打包规范、API 创建方式、挂载到执行环境的方法，以及运营最佳实践。

原文链接：https://developers.openai.com/blog/skills-in-openai-api

API Skills Agent

什么是 Skill？

Skill 是一个可复用的文件包（指令 + 脚本 + 资产），打包为一个文件夹，并以必需的 SKILL.md 清单文件为锚定。OpenAI 将该包复制到执行环境中，使模型可以按需读取指令并运行代码。

在托管 shell 中，当你将 Skills 附加到 shell 工具环境（environment.type="container_auto"）时：

  • 服务将 Skills 上传并解压到运行时
  • 服务读取 SKILL.md frontmatter（name/description），然后将每个 skill 的 name、description 和 path 添加到隐藏的系统提示上下文中
  • 如果模型决定调用某个 skill，它使用 path 读取 SKILL.md，然后通过 shell 工具探索文件并执行脚本

Skills 适用于流程：可重复的工作流，其中方式很重要（步骤、分支逻辑、格式规则、脚本）。

何时使用 Skills

Skills 在以下情况特别合适：

  1. 你想要可复用、独立版本化的行为集合。示例："PowerPoint 格式化程序"、"公司特定报告生成器"。
  2. 你的工作流高度条件化或分支复杂。如果 X → 做这个；否则如果 Y → 做那个；加上验证 + 重试。
  3. 你的工作流需要代码执行和本地产物。任何受益于脚本、模板、测试固件或应该与指令并置的参考资产。
  4. 你想保持系统提示精简。将稳定的流程放在 skills 中；系统提示保留全局行为。
  5. 多个 Agent 或团队共享同一"规范风格"。Skills 是一个不错的"组织标准库"模式。
  6. 你需要可复现性：Skills 天然兼容通过 skill 版本进行版本固定。

Skills vs. 工具 vs. 系统提示

层次用途避免
系统提示全局行为和约束、"始终做X"原则、小型稳定策略放置长篇多步骤流程
工具调用外部服务、创建副作用、获取实时状态范围过宽或副作用不明确
Skills可重复工作流、脚本/模板、沙箱中代码执行、按需调用一次性任务或每日变更的流程

Skill 打包：SKILL.md 和文件夹布局

Skill 只是一个文件夹包。示例结构：

  • SKILL.md（必需）
  • 脚本，如 *.py、*.js（可选）
  • 辅助工具和 requirements.txt
  • 资产、模板、示例输入

OpenAI 模型期望名称和描述来自 frontmatter（对于发现和路由很重要）。将 name 和 description 放在 SKILL.md frontmatter 中：

---
name: csv-insights
description: Summarize a CSV, compute basic stats, and produce a markdown report + a plot image.
---

# CSV Insights Skill

## When to use this
Use this skill when the user provides a CSV file and wants:
- a quick summary (row/col counts, missing values)
- basic numeric statistics
- a simple visualization

通过 API 创建 Skills

使用 POST /v1/skills 上传并验证你的 skill，从清单 frontmatter 中提取 name 和 description。

Option A：上传文件（multipart）

curl -X POST 'https://api.openai.com/v1/skills' \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -F 'files[]=@./csv_insights_skill/SKILL.md;filename=csv_insights_skill/SKILL.md;type=text/markdown' \
  -F 'files[]=@./csv_insights_skill/calculate.py;filename=csv_insights_skill/calculate.py;type=text/plain'

Option B：上传 zip

curl -X POST 'https://api.openai.com/v1/skills' \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -F 'files=@./csv_insights_skill.zip;type=application/zip'

如果遇到服务器错误，在本地打 zip 包并上传 zip。这是一个实用的变通方法。

将 Skills 挂载到执行环境

通过 Responses API 使用 skills，将其附加到 shell 工具的 tools[].environment.skills 中：

response = client.responses.create(
  model="gpt-5.2",
  tools=[{
    "type": "shell",
    "environment": {
      "type": "container_auto",
      "skills": [
        {"type": "skill_reference", "skill_id": "<skill_id>"},
        {"type": "skill_reference", "skill_id": "<skill_id>", "version": 2},
      ],
    },
  }],
  input="Use the skills to analyze the uploaded CSV and write outputs to /mnt/output."
)

运营最佳实践

1. 让 Skills 可发现

  • 在 frontmatter 中放置清晰的 name 和 description。
  • 在 SKILL.md 中包含：何时使用、如何运行、预期输出、注意事项。
  • 添加明确的路由指导："何时使用"与"何时不使用"，以及一些关键边缘情况。
  • 在正例旁边包含反例（skill 不应被触发时）以提高路由准确性。

2. 优先使用 zip 上传

Zip 包可移植、易于版本化，也是上传行为异常时的实用变通方法。

3. 在生产中固定版本

你要能说"运行此流程版本"，而不是"运行最新版本"。

  • 如何固定：version: 2
  • 如何浮动：version: "latest"
  • 省略时：默认为 default_version

4. 将 Skills 设计为小型 CLI

一个好的 skill 脚本：从命令行运行；打印确定性 stdout；遇到错误时大声失败并显示用法/错误；在需要时将输出写入已知文件路径。

5. 避免在系统提示中重复 Skills

如果系统提示重复了整个流程，会绕过 skills，失去可复用性 + 版本控制 + 条件调用的整个价值。

限制与验证

  • SKILL.md 匹配不区分大小写
  • 每个包只允许一个清单文件（skill.md/SKILL.md）
  • 最大 zip 上传大小：50 MB
  • 每个 skill 版本最多文件数：500
  • 最大未压缩文件大小：25 MB

结论

Skills 是提示词和工具之间缺失的"中间层"：提示词定义始终开启的行为，工具提供原子能力和副作用，而 Skills 打包可重复流程（指令 + 脚本 + 资产），模型可以按需挂载和执行。

使用 Skills 保持你的系统提示精简，让你的工作流持久耐用。从小处开始——用清晰的 SKILL.md 打包一个稳定流程，使其可作为小型 CLI 运行，然后发布。进入生产后，固定版本以获得可复现性，通过发布新版本安全迭代，并将你的 skills 库视为内部标准库：经过审计、可发现、在 Agent 间共享。
