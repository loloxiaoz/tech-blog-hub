Using Goals in Codex

2026-05-09约 6 分钟CodexGoals
Summary
Goals 是 Codex 中的持久化目标机制，让线程持续向定义好的结果推进。本文介绍如何编写带有可测量结果、验证方式和约束条件的 Goal，以及 /goal、/goal pause、/goal resume、/goal clear 等生命周期命令的使用方法。

Authors: Raj Pathak, Stefano Fabbri | Date: May 9, 2026 | Source: OpenAI Cookbook

Goals are persistent objectives in Codex that keep a thread working toward a defined outcome across turns. A Goal gives Codex a completion condition: what should be true, how success should be checked, and what constraints must stay intact.

A Goal is not background autonomy. It is a scoped, user-controlled completion contract.

What You Will Learn

By the end of this guide:

  • Decide when a Goal is better than a one-off prompt
  • Write a Goal with measurable outcome, verification surface, and constraints
  • Use /goal, /goal pause, /goal resume, /goal clear to manage lifecycle

Quickstart: Using Goals

Install or update Codex (Goals available starting Codex 0.128.0):

npm install -g @openai/codex@latest
codex --version

Set a Goal with /goal followed by the outcome:

/goal Reduce p95 latency below 120 ms without regressing correctness tests

Manage the lifecycle:

/goal        View current Goal
/goal pause  Pause an active Goal
/goal resume Resume a paused Goal
/goal clear  Remove current Goal

Goals vs Prompts

PromptGoal
ask → work → result → waitwork → check → continue or complete

Prompt: Do this next thing.

Goal: Keep working until this outcome is true.

A Goal gives Codex a finish line. The work still has to be audited against evidence.

How to Write a Goal

A good Goal defines six things:

  1. Outcome: What should be true when work is done
  2. Verification surface: Test, benchmark, report, or artifact that proves it
  3. Constraints: What must not regress while Codex works
  4. Boundaries: Which files, tools, or resources Codex may use
  5. Iteration policy: How Codex should decide next action after each attempt
  6. Blocked stop condition: When Codex should stop and report

Weak vs Strong Examples

Weak:

/goal Improve performance

Strong:

/goal Reduce p95 checkout latency below 120 ms, verified by checkout benchmark,
while keeping correctness suite green. Use only checkout service, benchmark
fixtures, and related tests. Between iterations, record what changed,
what benchmark showed, and next best experiment to try. If benchmark
cannot run or no valid paths remain, stop with attempted paths,
evidence gathered, the blocker, and next input needed.

What Changes When a Goal is Active

  1. Objective stays visible — attached to thread across turns
  2. Continuation becomes possible — Codex can continue from idle thread when Goal is active and within budget
  3. Completion must be evidence-based — not because model believes it's done

How Goals are Designed in Codex

Goals are implemented as persisted thread state, not global memory.

At architecture layer, a Goal is a durable, thread-scoped state recording:

  • Objective
  • Lifecycle (active/paused/complete/budget-limited)
  • Budget accounting
  • Progress accounting

Turning a Weak Goal into a Strong One

Example: Performance Tuning

Weak: /goal Improve performance

Strong: /goal Reduce p95 latency below 120 ms on checkout benchmark while keeping correctness tests green

Example: Fixing a Flaky Test

Weak: /goal Fix the failing checkout test

Strong: /goal Make checkout test suite pass on current branch without changing public API behavior. Verify by running full suite. If test is non-deterministic, document reproduction steps and recommended stability fix.

Example: Research Reproduction

Weak: /goal Reproduce the paper

Strong: /goal Produce strongest evidence-backed reproduction of Bühlmann et al., "Deep Hedging," using available paper materials and local resources. Attempt every headline result, verify outputs where possible, and end with report that separates confirmed findings, approximate reconstructions, blocked claims, and remaining uncertainty.

Using Goals for Complex Research

Research Goals should:

  • Decompose paper into claim inventory
  • Map claims to available evidence
  • Label blockers honestly (cannot exactly replay without original training state)
  • Produce audit that separates confirmed/approximate/blocked claims

When NOT to Use Goals

  • One-line edit, simple explanation, short code review → use normal prompt
  • Finish line is vague → do not use Goal
  • Want to hide uncertainty → do not use Goal (say so in the Goal)

Goals are strongest when task has three properties:

  1. Durable objective
  2. Evidence-based finish line
  3. Path that may require several turns of investigation

Continuation Rules

Codex checks for continuation only at safe boundaries:

  • After a turn has finished
  • When no other work is pending
  • When no user input is queued
  • When thread is idle

Conclusion: Let the Objective Persist, but Let Evidence Decide

Goals change the operating model of Codex: from sequence of isolated prompts into stateful work loop around defined outcome.

The architecture is intentionally bounded. The Goal is scoped to a thread, carries lifecycle state and budget accounting, and can be paused, resumed, cleared, completed, or stopped by budget.

For complex research, that is the difference between generating an answer and producing an auditable artifact.
