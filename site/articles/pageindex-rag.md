# PageIndex：一种基于推理的无向量 RAG 新范式

**来源**：知乎专栏，流光码梦者（西北工业大学，控制理论硕士），3 赞  
**原文**：https://zhuanlan.zhihu.com/p/2011422496105186987  
**注意**：赞数极低，测试数据来自 PageIndex 自家系统（Mafin 2.5），可信度需交叉验证

## 核心主张

传统向量 RAG 的根本缺陷是**相似性 ≠ 相关性**。PageIndex 用 LLM 推理导航替代向量近邻搜索，把文档索引放进 LLM 上下文窗口，实现结构感知的精准检索。

## 传统 RAG 的五大瓶颈

1. **查询意图 ≠ 内容相似**：问"如何优化 5G 功耗"，答案在"DRX机制"/"调度器设计"等章节——与问题语义不相似但逻辑高度相关
2. **语义相似 ≠ 真实相关**：同一词出现多处，向量检索全召回，但只有特定章节才是真正相关的
3. **硬分块破坏语义完整性**：固定 512 token 切断句子、段落、表格，跨块推理困难
4. **无法整合对话历史**：每次检索独立，多轮追问丢失上下文
5. **无法跟随文档内部引用**："详见第7.3节"——向量检索无法解析这种跨章节引用

## PageIndex 架构

### 离线阶段：构建 PageIndex 树

将文档解析为 JSON 层级树，每个节点含：
- `node_id`：唯一标识，映射到原始内容
- `title`：章节标题
- `summary`：LLM 生成的章节摘要（推理筛选的关键）
- `start_index` / `end_index`：文档位置
- `sub_nodes`：子节点数组

### 在线阶段：推理检索循环（5步）

1. 阅读顶层节点摘要，理解文档结构
2. 推理选择最可能包含答案的章节
3. 通过 `node_id` 获取原始内容
4. 判断信息是否足够 → 不足则继续探索，足够则进入生成
5. 整合所有已获取信息生成答案

索引驻留在 LLM **上下文窗口内**（In-Context Index），不用外部向量库。

## 测试数据（需谨慎）

| FinanceBench 准确率 | 98.7% | 72.3% | +26.4% |
| 多跳推理成功率 | 89.3% | 54.7% | +34.6% |
| 上下文利用率 | 92.1% | 47.6% | +44.5% |

## 当前局限

- OCR 依赖：复杂排版/扫描件构树准确率下降
- 领域迁移：摘要模型需针对新领域微调
- 不支持增量更新：文档更新需重建索引
