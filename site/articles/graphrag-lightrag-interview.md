# GraphRAG与LightRAG深度解析（面试向）

## 基本信息

- **形式**：面试题解析文章，覆盖 GraphRAG 和 LightRAG 核心原理
- **来源背景**：字节跳动二面真题整理

## 传统 RAG 三大痛点

### 痛点 1：跨文档多跳推理失败
场景示例："哪些欧洲供应商安全审计没通过且处理PII数据？"——需要同时关联供应商档案、审计报告、合同文档的**交集**。

传统 RAG 只能检索 Top-K 相似文本块，没有机制做集合交叉推理。

### 痛点 2：无法回答全局性问题
"这本小说主题是什么？" / "这份年报战略变化是什么？"——答案不在任何单一文本块中，需要**全局归纳**。

这类问题被微软称为 Query-Focused Summarization（QFS），本质是归纳任务不是检索任务。

### 痛点 3：Chunking 导致的语义断裂
- 语义被切断：完整论述横跨多段，切块后上下半段分离
- 实体关系丢失："卡托普利是ACEI药物"和"严重肾功能不全者禁用"被切到不同块，LLM 可能误解因果关系
- 检索噪声：语义相似但业务无关的块被召回（"苹果"的歧义问题）

**根本原因**：传统 RAG 做的是"找相似文本"，但企业需要"理解实体关系"。

## GraphRAG 核心原理

### 与传统 RAG 的区别
- 传统 RAG：知识以**离散文本块**存储，块间无关联
- GraphRAG：知识以**知识图谱**存储，实体节点 + 关系边 + 社区摘要

### 微软 GraphRAG（主流版本）
- 论文：《From Local to Global: A Graph RAG Approach to Query-Focused Summarization》（2024年4月）
- 核心：用 LLM 把文档"读成知识图谱"，再基于图谱检索和回答

### 工作流程
**离线阶段（图谱构建）**：
1. 文档切块（粗粒度）
2. LLM 抽取实体（节点）和关系（边）
3. 社区发现算法（Leiden算法）划分社区
4. LLM 为每个社区生成摘要

**在线阶段（检索问答）**：
- 全局查询：在社区摘要上并行生成部分答案 → 聚合为最终答案
- 局部查询：类似传统 RAG，但增加实体-关系路径追踪

### GraphRAG 的工程挑战

1. **Token 消耗巨大**：离线构建图谱成本极高（实体抽取、去重、社区摘要）
2. **实体质量参差不齐**：自动抽取的实体/关系含大量噪声，不适合作为可视化知识图谱
3. **增量更新困难**：新增文档时需要重新计算社区，成本高

## LightRAG 核心原理

### 与 GraphRAG 的主要区别
| 维度 | GraphRAG（微软） | LightRAG |
|------|----------------|---------|
| 社区摘要 | 需要（Leiden算法） | 不需要 |
| 索引层级 | 全局/局部两层 | 双层图（实体+关系） |
| 构建成本 | 高 | 低 |
| 适合场景 | 全局归纳型查询 | 局部+全局均衡型查询 |
| 增量更新 | 困难 | 相对容易 |

### LightRAG 的双层图
- **低层图（Low-level）**：具体实体和直接关系
- **高层图（High-level）**：抽象概念和主题级关系

## 场景选择指南

| 场景 | 推荐方案 |
|------|---------|
| 全局归纳（主题/趋势） | GraphRAG |
| 具体事实查询 | 传统 RAG |
| 多跳推理 + 成本敏感 | LightRAG |
| 代码库 / 结构化关系 | Graphify 类工具 |
| 企业合同/高审计要求 | 受控 Schema + 结构化字段 |
