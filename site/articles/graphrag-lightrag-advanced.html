核心论断

传统 RAG 的问题是"找不准"→ Advanced RAG 把检索策略调到最好 → 但有些问题不是找文本块能解决的 → GraphRAG 换了检索范式：从"找文本"到"找关系"。

三代 RAG 演进框架

代际核心定位主要优化
Naive RAG最原始 RAG文档切块 → 向量化 → 检索 → 生成；检索不准、幻觉严重
Advanced RAG优化检索策略混合检索、Rerank、Query改写、Parent-Child；把"找文本块"做到极致
GraphRAG换检索范式知识图谱 + 社区摘要预计算；从"找文本"到"找关系"

演进逻辑：Naive RAG 的问题是"找不准" → Advanced RAG 把检索策略调到最好 → 但有些问题不是找文本块能解决的 → GraphRAG 换了检索范式。

传统 RAG 的三个天花板

  1. 碎片化检索：chunk 是独立向量，跨 chunk 的关联信息无法同时召回（"张三在哪个项目，李四在哪个项目，他俩有没有合作"）
  2. 全局问题瞎答：Rerank/混合检索优化的是"找更相似的文本块"，拿到的还是碎片，只是碎片更多了
  3. 跨文档关系断裂：每个 chunk 独立 embedding，chunk 之间没有"关系"概念

GraphRAG 完整链路（6步）

索引阶段

Step 1：文档切块 和传统 RAG 一样，默认约 300 token，带 overlap。

Step 2：LLM 提取实体和关系（⚠️ 最烧钱的步骤） 每个文本块送给 LLM，识别实体（人/组织/地点/事件/概念）和关系。每个文本块都要过一遍 LLM。

文本块：["张三加入了项目X，负责后端架构设计"]
        ↓ LLM提取
实体：[张三(人), 项目X(项目)]
关系：[张三 → 负责后端架构设计 → 项目X]

Step 3：实体去重与合并 "Apple"和"Apple Inc."是同一实体，合并为一个节点。

Step 4：构建知识图谱 实体 = 节点，关系 = 边。

Step 5：Leiden 社区检测 用 Leiden 算法（Louvain 的改进版）对图谱做社区划分，产生层级式社区结构（底层小社区 → 上层大社区 → 顶层整图）。

Leiden vs Louvain：Louvain 可能产生"内部不连通"的社区，Leiden 保证社区内部一定连通，对节点连接不均匀的知识图谱更重要。

Step 6：生成社区摘要 对每个层级的每个社区，LLM 生成结构化摘要（community report），描述关键实体、核心关系、主要主题。

索引阶段最终产出：实体表、关系表、社区表（含摘要）、文本块向量索引。

查询阶段（三种方式）

查询方式机制适合场景
Local Search提取关键实体 → 图谱遍历邻居 → 聚合上下文 → LLM生成具体问题："张三在项目X里负责什么？"
Global SearchMap-Reduce：所有社区摘要分组 → 各组生成部分答案 → 汇总生成最终答案全局问题："公司所有项目技术栈趋势是什么？"
DRIFT (2024-10)先用社区摘要全局预判，再沿图谱局部深挖兼顾成本和深度（Global 和 Local 的折中）
Global Search 层级选择：选低层级社区 → 答案更细致；选高层级社区 → 答案更概括。

GraphRAG 落地三个坑（含具体数字）

坑一：建图太贵

  • 100 万 token 原始文档 → 索引阶段消耗 200-500 万 token
  • 索引成本是传统 RAG（只需 Embedding 一次）的 10 倍以上
  • 百万文档索引耗时约 3.8 小时

坑二：增量更新难

新增 10% 的文档，可能需要重跑 30-50% 的索引流程（社区检测 + 社区摘要重新生成）。原因：社区是全局结构，局部变更会引发全局调整。

DRIFT 搜索模式改进的是查询策略，不解决增量更新问题。

坑三：社区粒度难定

层级太高（社区太大）→ 摘要太笼统，细节丢光 层级太低（社区太小）→ 社区数量爆炸，Map-Reduce token 成本飙升

实际操作需要根据数据量和查询需求反复调试，没有公式可以直接算出。

LightRAG 双层图谱详解

底层：实体层图谱（管具体查询）

  • 节点 = 实体（人/组织/概念）
  • 边 = 实体之间的直接关系
  • 每个节点存：实体名称、类型、描述、来源文本块

上层：关系层图谱（管全局查询）

  • 节点 = 关键关系/交互（不是实体本身，这是与常规图谱的关键区别）
  • 边 = 关系之间的连接（因果、时序、主题关联）
  • 捕捉跨实体的模式，不需要预计算社区摘要

两层都带向量嵌入，检索时走"图遍历 + 向量相似度"混合方式。

四种查询模式

模式机制适合场景
Naive纯向量检索简单事实查询
Local实体层图谱遍历邻居具体问题（"张三做什么"）
Global关系层图谱找主题模式全局问题（"核心趋势"）
HybridLocal + Global 合并兼顾细节和全局（推荐默认）

增量插入机制（LightRAG 的杀手锏）

三层实体匹配：

  1. 名称精确匹配
  2. 名称模糊匹配 + 描述向量相似度
  3. 模糊时 LLM 辅助判断

匹配上 → 合并描述；未匹配上 → 新增节点/边；只更新受影响的向量嵌入。

复杂度：O(局部变更) vs GraphRAG O(整个图谱)

重要澄清：GraphRAG ≠ RAG + 知识图谱

面试高频踩坑问题。

加一个知识图谱做辅助检索确实能提升效果，但 GraphRAG 的核心创新不在"有图谱"，而在社区摘要的预计算。

GraphRAG = 知识图谱 + 社区层级 + 预计算摘要 + Map-Reduce 查询，四个缺一不可。

没有社区层级和预计算摘要，全局性问题还是答不好。

选型一句话原则

要深度选 GraphRAG，要灵活选 LightRAG。
  • 选 GraphRAG：知识库大且稳定、全局洞察是刚需、预算充足、更新频率低（典型：律所、药企、情报分析）
  • 选 LightRAG：知识库动态更新、成本敏感、快速上线、查询类型混合（典型：新闻聚合、客服知识库、个人论文库）
