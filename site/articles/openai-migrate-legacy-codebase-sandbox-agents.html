使用沙箱 Agent 迁移遗留代码库

2026-04-07 约 12 分钟 Agent Sandbox Code Migration
摘要
本教程介绍如何使用 Agents SDK 和沙箱技术迁移遗留代码库。核心模式是"沙箱即工具"：编排层运行在可信主机进程中，shell命令和文件编辑在隔离的沙箱中执行。支持 Docker、E2B、Cloudflare 等多种沙箱后端，可无缝切换而无需修改 Agent 代码。

原文链接：https://cookbook.openai.com/examples/agents_sdk/sandboxed-code-migration/sandboxed_code_migration_agent

概述

代码现代化是一项持续的工作。过时的依赖、安全风险、合规压力、遗留模式会在大型代码库中不断累积，一次性提交的大规模迁移PR不仅难以审核，合并风险也很高。

代码迁移Agent应当在受控环境中运行，每次只处理一个范围明确的任务：检查相关仓库、编辑文件、运行检查、返回补丁。

本教程使用 Agents SDK，将执行编排放在可信的主机进程中，而shell命令和文件编辑则在隔离的执行环境中运行。这种分离设计让主机编排可以使用密钥、工具、外部服务，同时只向沙箱提供当前任务所需的文件和命令。

完成本教程后，你将能够：

  • 让Agent编排层运行在执行环境之外，执行环境仅负责运行shell命令和文件编辑
  • 将现代化工作拆分为适合任务粒度的仓库分片
  • 通过测试、检查、产物和审计日志验证每个分片
  • 无需重写Agent即可切换沙箱提供商

架构设计：沙箱即工具

核心模式：可信主机进程负责Agents SDK编排层、工具、MCP服务器、凭证、策略、审计工作；执行环境仅接收当前任务的范围化工作区，以及运行命令和编辑文件所需的沙箱能力。

在本模式的流程中，沙箱是编排层在Agent需要文件系统、终端命令、测试运行或补丁时调用的工具，更完整的Agent栈都留在编排层一侧。

整体流程

  1. 你的应用收到迁移请求，将其拆分为任务粒度的仓库分片
  2. 对每个任务，主机侧的Agents SDK编排层启动一个Agent，创建一个全新的沙箱
  3. 编排层将该任务的仓库和迁移说明暂存到沙箱中
  4. Agent使用沙箱工具检查文件、编辑代码、运行测试
  5. 主机接收任务的报告和补丁，写入审计日志，删除沙箱，继续处理下一个任务

前置要求

  • Python 3.10+
  • 本地运行的Docker（用于Docker沙箱示例）
  • OpenAI API密钥，导出为 OPENAI_API_KEY 环境变量
  • 支持沙箱的OpenAI Agents SDK
  • 可选：托管沙箱提供商凭证，例如E2B的 E2B_API_KEY

步骤1：定义迁移任务

本教程在 repo_fixtures/ 目录下包含两个小型示例仓库。任务列表将每个迁移分片指向本地仓库：

@dataclass(frozen=True)
class MigrationTask:
    name: str
    repo_path: Path

    @property
    def migration_brief_path(self) -> Path:
        return self.repo_path / "MIGRATION.md"


DEFAULT_MIGRATION_TASKS = (
    MigrationTask(
        name="support_reply_service",
        repo_path=EXAMPLE_ROOT / "repo_fixtures" / "support_reply_service",
    ),
    MigrationTask(
        name="case_summary_service",
        repo_path=EXAMPLE_ROOT / "repo_fixtures" / "case_summary_service",
    ),
)

适配你自己的代码库时，只需替换任务的 repo_path，编辑对应的 MIGRATION.md 即可，通用的运行提示词可以保持不变。

步骤2：在Agent编辑前验证基线

在Agent修改仓库前，先自己运行相同的测试。Agent会在修改任何文件前，在该任务的沙箱内运行这个基线测试命令：

baseline = subprocess.run(
    [sys.executable, "-m", "unittest", "discover", "-s", "tests", "-t", "."],
    cwd=task.repo_path,
    text=True,
    capture_output=True,
    check=False,
)
assert baseline.returncode == 0

步骤3：暂存沙箱工作区

清单是沙箱的边界，它告诉沙箱客户端哪些主机文件需要暂存，以及它们在执行环境中的路径。对于真实的迁移场景，只暂存目标检出代码、任务说明和该次运行所需的文件，将凭证、客户存储、内存保留在主机编排层中。

def build_manifest(task: MigrationTask | None = None) -> Manifest:
    task = task or DEFAULT_MIGRATION_TASKS[0]
    return Manifest(
        root="/workspace",
        entries={
            "migration_agent/AGENTS.md": LocalFile(
                src=EXAMPLE_ROOT / "migration_agent" / "AGENTS.md"
            ),
            "repo": LocalDir(src=task.repo_path),
        },
    )

步骤4：定义沙箱Agent

Agent获得两个沙箱能力：Shell()（用于终端操作）和 ApplyPatch()（用于文件编辑）。定义中其余部分都保留在主机编排层：指令、模型设置、MCP服务器、类型化输出契约。

def build_agent(*, model: str, manifest: Manifest, mcp_servers=None) -> SandboxAgent:
    return SandboxAgent(
        name="Code Migration Agent",
        model=model,
        instructions=AGENT_INSTRUCTIONS,
        developer_instructions=DEVELOPER_INSTRUCTIONS,
        default_manifest=manifest,
        capabilities=[Shell(), ApplyPatch()],
        mcp_servers=list(mcp_servers or []),
        model_settings=ModelSettings(tool_choice="required"),
        output_type=MigrationResult,
    )

agent = build_agent(model="gpt-5.4", manifest=manifest)

步骤5：运行迁移任务流

完整的运行是主机侧对迁移任务的循环：对每个任务，编排层构建清单和Agent，创建一个全新的沙箱会话，将会话传入 Runner.run_streamed。任务完成后，主机将返回的补丁包写入 outputs/<task_name>/ 目录，删除沙箱，再开始下一个任务。

步骤7：切换沙箱提供商

本部分展示三种沙箱后端，模式完全相同：修改沙箱客户端，无需修改Agent。

Docker（本地运行）

client = DockerSandboxClient(docker.from_env())
session = await client.create(
    manifest=manifest,
    options=DockerSandboxClientOptions(image="python:3.14-slim"),
)

E2B（托管沙箱）

export E2B_API_KEY="..."
python src/run_migration_agent.py --backend e2b

Cloudflare（托管Worker沙箱）

export CLOUDFLARE_SANDBOX_WORKER_URL="https://..."
python src/run_migration_agent.py --backend cloudflare

生产环境注意事项

边界生产模式
编排层将编排、工具、凭证、策略、审计保留在主机中
沙箱仅暂存任务工作区，在其中运行命令和编辑，任务完成后销毁
数据访问通过主机路由客户存储和网络访问，不要直接暴露给沙箱
输出在主机中验证沙箱输出，再展示给用户或应用变更

后续步骤

适配本模式时，将 migration_tasks 替换为你的仓库、包或服务，为每个任务分配一个检出代码和一个 MIGRATION.md，保持验证命令明确，返回补丁供审核而不是直接自动应用主机侧变更。添加确定性评估，检查迁移契约是否满足，而不仅仅是测试套件是否通过。
