使用 Codex 构建迭代修复循环

2026-05-11约 10 分钟CodexAgent迭代修复
Summary
介绍如何使用 Codex CLI 构建闭环智能体工作流：审查→修复→验证三阶段迭代，以文档可靠性修复为例，展示结构化输出交接、并发处理、迭代收敛和审计跟踪的完整实现方案。
原文链接
https://cookbook.openai.com/examples/codex/Build_iterative_repair_loops_with_Codex

概述

本教程介绍闭环智能体工作流：智能体先生成输出，再验证输出，最后根据反馈优化下一轮的输出。

我们将以文档可靠性工作流为例：检测、修复、验证过时或损坏的API和SDK示例。示例使用了本Cookbook仓库中适配的故意过时的notebook文件。

我们将使用Codex构建这个智能体循环：Codex会检查当前状态、应用针对性修改、运行验证，当反馈显示仍有问题时重复上述步骤。

注意：notebook任务仅为示例，该模式适用于所有能够用可靠反馈衡量智能体输出的场景。

整个工作流分为三个阶段：

  1. 审查：检查当前工件，返回结构化发现，不编辑文件
  2. 修复：根据发现和最新验证反馈，对复制的工件应用针对性修改
  3. 验证：运行相关检查，报告仍需要优化的内容

验证是闭环的关键：修复后的notebook必须满足相关检查要求，所有遗留问题都会成为下一轮修复的输入。

环境配置

本notebook使用无头模式的Codex CLI，因此修复步骤可以直接在Python单元格中运行，无需使用聊天UI。第一个代码单元格会安装CLI，如果你已经安装了CLI，可以跳过该单元格。

在运行实时修复循环之前，需要在环境中设置OPENAI_API_KEY。

本notebook默认使用快速修复模型，确保完整示例可以在合理时间内运行完毕。如果要尝试其他模型，可以在开始前设置REPAIR_MODEL。安装单元格固定了已知的Codex CLI版本以确保可复现性，当你需要更新的CLI行为时可以手动更新版本。

!npm install -g @openai/codex@0.130.0
MODEL = os.getenv("REPAIR_MODEL", "gpt-5.4-mini")

加载示例工件

以下单元格会加载三个配套notebook，并汇总驱动修复循环的元数据。

示例特意做得很小，运行速度快，但仍然可以验证架构：审查阶段会发现实质性问题，修复阶段会做针对性修改，验证阶段会为下一轮生成反馈。

在本示例中，验证会端到端执行每个修复后的notebook。如果是其他领域，验证也可以是单元测试、策略检查、schema 验证器、模拟仿真或者人工审批步骤。关键是失败要转化为结构化反馈，而不是死路一条。

定义业务规则和问题分类

在让Codex审查或修复工件之前，需要先给它一个小的共享约定，这样可以让循环聚焦于重要问题，而不是让模型从零开始推断所有产品和风格规则。

以下规则定义了这些示例notebook的"合格"标准：使用最新的API模式、明确的设置步骤、可本地运行的示例、保留原始教学目标。

审查阶段

审查阶段读取工件并返回结构化发现，不运行验证，也不编辑文件。这种分离让第一步保持聚焦：在修改任何内容之前先识别可能的问题。

我们将审查提示发送给codex exec并附带JSON schema，schema可以保证结果可被机器读取，后续单元格可以直接将发现项传入修复提示，无需从之前的回答中抓取文本。

修复阶段

修复阶段会获取当前工件、审查发现、业务规则，以及上一轮的任何验证反馈。随着循环不断学习，提示会变得更加具体。

Codex会在迭代目录中编辑副本，并返回修改内容的简短摘要。循环不会假设修改已经生效，验证阶段会在下一步判断修改效果。

验证阶段

验证的工作方式类似于小型评估：我们定义期望的行为，运行相关检查，然后让判定器根据评判标准对结果打分。

对于文档示例，首先执行notebook。很多notebook问题只有在运行时才会出现：缺少导入、过时的文件路径、依赖旧API响应的单元格，或者对作者来说清晰但对新读者来说不清晰的设置指南。

如果验证失败，失败信息会成为下一轮修复传递的证据，这样可以让下一轮修复基于观察到的行为，而不是仅仅依赖diff中看起来正确的内容。

运行迭代循环

每个notebook案例都是独立的，因此我们可以并发处理这些案例，这样既可以让演示快速运行，又能为每个示例保留相同的审查、修复和验证流程。

迭代结果示例：

  • 第1轮：最简单的固定装置通过；更深的固定装置保留小差异
  • 第2轮：中等深度的固定装置在看到验证反馈后清除
  • 第3轮：最深的固定装置清除或留下聚焦的最终差异

摘要告诉我们什么

重要的信号不是Codex做了修改，而是随着循环运行，剩余的验证差异越来越小。

迭代信号重要性
第1轮最简单的固定装置通过；更深的固定装置保留小差异循环可以做初始修复，同时传递仍需要证据的案例
第2轮中等深度的固定装置在看到验证反馈后清除运行时和判定器反馈成为有用的修复指令
第3轮最深的固定装置清除或留下聚焦的最终差异循环收敛，或者产生清晰的交接给人工审查者

record.json文件是让这个过程可审计的地方。有用的记录可以回答四个问题：审查发现了什么、Codex修改了什么、notebook是否执行、还剩什么？

泛化到连续循环

上面的固定三轮运行对于教授模式很有用。生产循环应该自己决定何时停止。

一个好的循环通常会因为四个原因之一停止：验证通过、循环达到最大尝试次数、剩余差异停止变化、或者下一个决策需要人工审查。这些停止条件和修复提示一样重要。

另一个生产细节是审计跟踪。保存每轮的审查发现、修复的工件、验证结果、验证判定和剩余差异。

其他适用场景

notebook演练只是教授架构的一种方式。只要智能体更改的文件或流程在被接受之前需要不止主观审查，同样的模式都能提供帮助。

一些高价值示例：

  • 协议优化：起草更新供专家审查，然后根据剂量规则、时间限制或所需的安全检查进行验证
  • 监管修复：起草受监管内容的更新，然后检查所需语言、引用、批准和特定司法管辖区的术语是否仍然完整
  • 支持知识刷新：更新文章，根据当前产品行为或已知解决方案进行测试，并将不匹配项传递到下一轮
  • 代码现代化：替换已弃用的API，运行测试或静态检查，使用剩余失败指导下一个修复

结论

迭代修复循环使智能体维护更容易审查和运作，因为它们将判断与证明分开。

审查阶段发现候选问题，修复阶段进行针对性修改，验证阶段执行工件并产生下一个差异。当这些阶段交换结构化输出时，工作流变得更容易检查、重复和适应。

主要思想很简单：不要依赖单次传递，而是给工作流一种从工件中学习、进行有界修复、并对真实验证反馈做出反应的方法。这个小改变让智能体维护更加实用。
