# 阿里Agent岗二面：RAG检索优化四层系统性框架

## 基本信息

- **性质**：大厂面试题解析，RAG 检索优化的系统性视角
- **核心论点**：检索层是 RAG 投入产出比最高的优化环节，需要系统性四层框架而非零散调参

## 核心洞见

> 检索是 RAG 的命脉。LLM 只能根据送进去的 context 来回答，检索召回的内容就是整个系统的天花板。

**四层优化框架（递进关系）**：
```
索引层   → 仓库里放了什么（知识怎么存）
查询层   → 用什么钥匙开门（问题怎么转换）
召回层   → 从哪几扇门进去找（用什么路径检索）
重排序层 → 把找到的最好的几件带出来（精排）
```

## 第一层：索引优化（Small-to-Big）

**核心矛盾**：检索需要小粒度（语义聚焦，向量精准），LLM 需要大粒度（上下文完整）——两者天然冲突。

**三种解决方案**：

| 方案 | 机制 | 特点 |
|------|------|------|
| **Parent-Child Chunking** | 子 chunk 建索引（150 token），命中后取父 chunk（500 token）给 LLM | 最直接，效果稳定 |
| **Summary Index（摘要索引）** | LLM 为每段生成摘要，用摘要向量检索，命中后返回原文 | 摘要语义更聚焦，命中率更高 |
| **多粒度分层索引** | 同时建章节级/段落级/句子级三层索引，按问题类型自动选粒度 | 覆盖更多问题类型 |

## 第二层：查询优化

**背景问题**：用户口语化表达 vs 知识库书面语表述，即使语义相近，向量距离也可能较远。

| 方法 | 机制 | 适用场景 |
|------|------|---------|
| **Query 改写** | LLM 将口语/含歧义问题转为正式精准表达 | 用户表述质量差 |
| **Multi-Query 扩展** | 将一个问题扩展为 3-5 个不同角度的问法，并行检索合并 | 问法和文档描述角度对不上 |
| **HyDE** | LLM 先生成假设答案，用假设答案向量检索（文档-文档距离比问题-文档更近） | 知识库领域明确时效果稳定 |
| **Step-back Prompting** | 将具体问题往上抽象一层，先检索通用背景知识，再结合背景回答具体问题 | 知识库中只有通用背景无直接答案 |

> **注意**：Multi-Query 时，原始问题必须保留在检索列表里，改写过程可能丢失原始细节。

## 第三层：召回优化（多路并行+RRF）

**单路召回的根本局限**：
- 向量检索：擅长语义，但精确词语匹配（如产品型号、编号）效果差
- BM25 关键词：精确匹配强，但无法处理同义表达

**三路并行典型方案**：向量检索 + BM25 关键词检索 + 知识图谱检索（可选）

**RRF 融合公式**：
```
score(chunk) = Σ(1 / (k + rank_i))
```
- `k = 60`（经验最优，加"保底分"不让排名靠后的候选被完全淘汰）
- 只看排名不看原始分数，解决了各路分数量纲不同的问题

## 第四层：Rerank（Cross-encoder 精排）

**问题**：多路召回后有 20-30 个候选，全部塞给 LLM 会导致：
1. Token 成本暴涨
2. **Lost in the Middle** 现象：LLM 处理长文本时只关注开头和结尾，中间内容被忽略

**Bi-encoder vs Cross-encoder**：

| 维度 | Bi-encoder（向量检索） | Cross-encoder（Rerank） |
|------|---------------------|----------------------|
| 输入方式 | Query 和 Doc 分别编码 | Query + Doc 拼接，同时输入 |
| 相关性判断 | 无法看到词语级别的交互 | 每个词都能与对方词语交互 |
| 速度 | 极快（向量预计算） | 慢（每对都要推理） |
| 精度 | 中 | 高 |
| 类比 | 只看简历匹配 | 面对面面试 |

**推荐候选集设置**：先检索 Top-50~100，Rerank 后取 Top-3~5 送入 LLM。

## 典型生产级组合

```
Parent-Child 索引 + 向量/BM25 多路召回 + Rerank 精排
```
（Query 改写按需加，在用户表述质量差时额外启用）
