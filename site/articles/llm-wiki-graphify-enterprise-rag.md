# Karpathy的LLM Wiki + Graphify：企业级 RAG 缺的真是知识图谱？

## 基本信息

- **来源**：知乎专栏《企业RAG从入门到落地》
- **核心方法**：3套方案 × 30份合成合同 × 48道测试题的对比实验

## 核心主张（一句话）

个人知识库的最佳实践不等于企业知识库的最佳实践。企业合同知识库真正缺的不是酷炫工具，而是**受控schema、结构化字段、原文引用、权限边界和可复跑评测**。

## 三大方案实测对比

### 方案 A：基础向量 RAG（baseline）
- 测试口径：30份合同，24题
- Answer Accuracy：6/24，**25%**
- 表现：简单事实题（5/5 全对），跨合同综合、字段筛选全失败

### 方案 B：LLM Wiki 模式（5份样本验证）
- 测试口径：5份合同 demo，10题
- Answer Accuracy：10/10，**100%**
- 表现：跨文档综合问题显著改善，但有口径限制（仅小范围预编译知识）

### 方案 C：受控 schema + 关系表 + 原文索引（综合方案）
- 测试口径：30份合同，24题
- Answer Accuracy：24/24，**100%**
- 设计：SQLite 数据库（合同表、项目表、关系表、风险事件表）+ ChromaDB（向量检索）
- 核心逻辑：**先用结构化字段确定范围，再用向量库回原文**

## LLM Wiki 核心概念

### 与传统 RAG 的区别
| 维度 | 传统 RAG | LLM Wiki |
|------|---------|---------|
| 知识处理时机 | 提问时临时检索 | 资料进入时先加工 |
| 知识形态 | 文档 chunk | 摘要页、实体页、综合页、索引 |
| 多文档综合 | 每次重新拼 | 提前沉淀成综合页 |
| 矛盾处理 | 靠提问时发现 | ingest/lint 时提前标记 |

### 企业级降维用法（关键洞见）
不要全量编译，只对**高频复杂问题**提前生成综合页：
- 触发条件：某类问题反复出现
- 生成方式：LLM 初稿 + 规则校验 + 必要时人工抽查
- 引用要求：每个关键结论都要能指回原始合同或结构化字段

## Graphify 评价

- **代码场景**：可以帮助摸底代码库结构（核心抽象如 Client/Response），但全量图往往信息过载
- **合同场景**：不能直接用，需要受控 schema 兜底（合同关系是业务约束，不是语法树关系）
- **71.5x token reduction** 数字仅代表上下文压缩率，不代表准确率提升

## 企业合同知识库五层架构

| 层级 | 作用 | 关键要求 |
|------|------|---------|
| 文件解析 | 解析 Word/PDF | 保留页码、标题、表格位置 |
| 合同结构化 | 抽取关键字段 | 字段固定，结果入库 |
| 条款索引+受控图谱 | 条款级 chunk + schema 关系 | 关系必须有来源 |
| 问答层 | 权限过滤、混合检索 | 答案必须引用原文 |
| 评测与审计 | 测试集、引用检查 | 用结果反推系统问题 |
