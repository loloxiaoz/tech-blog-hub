Codex Prompting Guide

2026-02-25 约 15 分钟
CodexPromptingAgent
本指南介绍如何为 Codex 模型（gpt-5.3-codex）优化提示词和工具配置。涵盖推荐的基础提示词、自主性设置、代码实现规范、工具使用指南（apply_patch、shell_command、update_plan）、Preamble 机制，以及新版 gpt-5.3-codex 的 Phase 参数和 Persona 设置。
原文链接：https://developers.openai.com/blog/codex-prompting-guide

简介

Codex 模型是当前编码 Agent 的最先进模型。本指南适用于通过 API 直接使用模型以获得最大可定制性；我们也提供了 Codex SDK 用于更简单的集成。

API 中提供的调优模型为 gpt-5.3-codex，详见 模型页面。

Codex 模型近期改进

  • 更快、更高 token 效率：使用更少推理 token 完成任务。推荐使用 medium 推理努力度作为交互式编码的均衡快速模型。
  • 更高智能和更长自主运行：Codex 高度能力化，可以在你最难的任务上自主运行数小时。可使用 high 或 xhigh 推理努力度处理最困难的任务。
  • 原生压缩支持：压缩功能使多小时推理不会触达上下文限制，也让用户对话无需开启新的聊天会话即可持续更长时间。
  • 更好的 PowerShell 和 Windows 环境支持：Codex 现在在 PowerShell 和 Windows 环境中更加强大。

迁移到 codex-cli 的关键步骤

如果你已有可用的 Codex 实现，模型应该只需最少修改即可正常工作；但如果你为 GPT-5 系列模型或第三方模型优化了提示词和工具集，我们建议进行更显著的调整。最佳参考实现是我们完全开源的 codex-cli agent。

  1. 更新你的提示词：如果可能，以我们的标准 Codex-Max 提示词作为基础，并从中进行有针对性的补充。最关键的片段是涵盖自主性、持续性、代码库探索、工具使用和前端质量的部分。
  2. 更新你的工具：包括我们的 apply_patch 实现和以下最佳实践。这是获得最佳性能的主要杠杆。

推荐基础提示词

此提示词最初是默认的 GPT-5.1-Codex-Max 提示词，后来针对答案正确性、完整性、质量、正确的工具使用和并行性以及行动偏向进行了进一步优化。

You are Codex, based on GPT-5. You are running as a coding agent in the Codex CLI on a user's computer.

# General
- When searching for text or files, prefer using `rg` or `rg --files` respectively.
- If a tool exists for an action, prefer to use the tool instead of shell commands.
- When multiple tool calls can be parallelized, use make these tool calls in parallel.
- Default expectation: deliver working code, not just a plan.

# Autonomy and Persistence
- You are autonomous senior engineer: once the user gives a direction, proactively gather context, plan, implement, test, and refine without waiting for additional prompts.
- Persist until the task is fully handled end-to-end within the current turn.
- Bias to action: default to implementing with reasonable assumptions.

# Code Implementation
- Act as a discerning engineer: optimize for correctness, clarity, and reliability.
- Conform to the codebase conventions: follow existing patterns, helpers, naming, formatting.
- Keep type safety: Changes should always pass build and type-check.

In-progress 用户更新

Codex 模型系列可以在工作过程中输出进度更新。对于 gpt-5.3-codex 之前的版本，这些更新是系统生成的，无法通过提示词控制。对于 gpt-5.3-codex 及之后的版本，这些更新更具沟通性，并将提供更多关于正在发生的事情及原因的关键信息。

使用 AGENTS.md

codex-cli 自动枚举这些文件并将它们注入对话中；模型经过训练会严格遵循这些指令。

  1. 文件来自 ~/.codex 以及从仓库根目录到当前工作目录的每个目录。
  2. 它们按顺序合并，后面目录中的文件会覆盖前面的。
  3. 每个合并块以独立的用户角色消息形式出现。

Context Compaction

Compaction 显著扩大了有效上下文窗口，用户对话可以进行多轮而不会触达上下文窗口限制或出现长上下文性能降级，Agent 也可以执行比典型上下文窗口更长的轨迹。

工作原理

  1. 你照常使用 Responses API，发送包含工具调用、用户输入和助手消息的输入项。
  2. 当你的上下文窗口变大时，你可以调用 /compact 生成一个新的压缩上下文窗口。
  3. 对于后续调用 /responses 端点，你可以传入新的压缩对话项列表。模型将保留关键的先前状态，同时使用更少的对话 token。

工具使用指南

  1. 我们强烈推荐使用精确的 apply_patch 实现，因为模型在此 diff 格式上经过高度训练。
  2. 对于终端命令，我们推荐使用我们的 shell 工具。
  3. 对于计划/TODO项，我们的 update_plan 工具效果最好。

Apply_patch 工具

实现 apply_patch 最简单的方式是使用 Responses API 中我们的原生实现，你也可以使用带 context-free grammar 的 freeform 工具实现：

from openai import OpenAI
client = OpenAI()

tools = [
    read_file_tool,
    cast(ToolParam, {"type": "apply_patch"}),
]

response = client.responses.create(
    model="gpt-5.1-Codex-Max",
    input=input_items,
    tools=tools,
    parallel_tool_calls=False,
)

Shell_command 工具

这是我们的默认 shell 工具。注意我们发现使用字符串类型的命令比命令列表性能更好：

{
  "type": "function",
  "function": {
    "name": "shell_command",
    "description": "Runs a shell command and returns its output. Always set the workdir param.",
    "parameters": {
      "type": "object",
      "properties": {
        "command": {"type": "string"},
        "workdir": {"type": "string"},
        "timeout_ms": {"type": "number"}
      },
      "required": ["command"]
    }
  }
}

工具响应截断

我们建议按以下方式截断工具调用响应：

  • 上限为 10k token。可以通过计算 num_bytes / 4 粗略估算。
  • 如果达到上限，应使用一半预算用于开头，另一半用于结尾，中间截断为 ...{N} tokens truncated...。

GPT-5.3 Codex 新特性

Phase 参数

为更好地支持 gpt-5.3-codex 的 preamble 消息，Responses API 包含了 phase 字段，旨在防止长时运行任务期间的提前停止和其他不良行为。

phase 的值为：

  • null
  • "commentary"：对应助手消息应被视为评论/preamble内容
  • "final_answer"：对应助手消息应被视为最终、结论性内容

重要：正确保留助手项上的 phase 是 gpt-5.3-codex 的要求。如果助手 phase 元数据在历史重建过程中丢失，可能导致显著的性能降级。

Persona（暖色 vs 稳重）

Codex 应用和 CLI 支持两种 Persona：

  • Warm（暖色）：更多温暖感、伙伴式配对能量。更多致谢、安慰和情境设置。适合用户从叙事导向中受益的场景（入职、模糊任务、高风险变更）。
  • Grounded（稳重）：更简洁、直接、以交付为导向。每 token 可操作信息更多。适合延迟/吞吐量重要，或用户熟悉工作流程并只想要进度和结果时。

故障排除与元提示

我们明确追踪的常见失败模式：

  • 过度思考/在第一个有用动作之前延迟过长
  • 不自然、类似日志的状态更新，而不是配对编程协作
  • 尴尬的 preamble 措辞和重复性咒语（"Good catch"、"Aha"、"Got it–" 等）

上述失败模式通常可以通过元提示解决。你可以在轮次结束时询问模型如何改进其自身指令以应对表现不佳的响应。
