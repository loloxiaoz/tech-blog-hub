# 从 1600+ 份 Word 文档到生产级 RAG：工控行业知识库全链路实战复盘

## 基本信息

- **客户画像**：工控组态软件厂商，20+ 工程师，4 万多家存量客户，微信群答疑为主
- **核心成果**：两周内从数据清洗到系统交付，达到生产可用水平
- **最终架构**：FastAPI（后端）+ Next.js（前端）+ ChromaDB（向量库）+ Supabase（关系库）+ MinIO（图片存储）

## 核心论点

数据质量是决定 RAG 效果的关键因素，数据工程占整个项目约 **70% 工作量**。

> "企业 RAG 不是框架选型问题，是数据处理问题。"

## 数据工程要点

### 噪音类型（论坛爬取文档特征）
- 页眉页脚重复文字
- 上传时间戳（"2024-03-15 14:30 上传"）
- 下载统计（"(1.09 MB, 下载次数: 118)"）
- 论坛交互元素（"点击文件名下载附件"）
- 代码块按钮残留（"复制代码"）
- 用户抱怨性内容（"文件下载不了"）

### 双重输出设计
| 输出类型 | 格式 | 用途 |
|---------|------|------|
| 机器侧 | Clean Markdown | 向量化入库，RAG 检索 |
| 用户侧 | Clean DOCX | 用户点击引用时跳转查看 |

### 富语义文本块
每个切片开头附带：文档标题 + 摘要 + 关键词，解决 LLM 看到碎片无法理解背景的问题。

### LLM 元数据增强
为每份文档生成：
- 一句话摘要（summary）
- 5-8 个关键技术术语（keywords）
- 3-5 个文档能回答的问题（questions）— 配合 HyDE 提升召回率

## 检索优化迭代过程

| 方案 | Recall@5 | 平均延迟 | 成本 |
|------|---------|---------|------|
| Baseline（无 Rerank, Top-5） | 96.7% | ~0.1s | 0 |
| Rerank API（Top-50） | 100% | ~7s（不稳定） | 按 token |
| Rerank 本地（Top-50） | 100% | ~12.6s | 0 |
| **Rerank 本地（Top-20）** | **100%** | **~2.8s** | **0** |

**关键决策**：粗排候选从 50 降到 20，延迟从 12s 降到 3s，召回率保持 100%。
这是"工程甜点"：评测数据支撑的工程决策，而非拍脑袋。

### 父子连坐召回（核心优化）
问题：Chunk 0（含摘要）"霸榜" Top-K，真正有答案的正文切片被压到排名靠后。
解法：只要检索到任意一个切片，就强制召回该文档的**全部切片**，重构完整上下文。

## 数据飞轮设计

### 反馈机制
- 点赞：直接入库
- 点踩：弹出输入框要求填写原因（定位是检索层问题还是生成层问题）

### 为什么不一上来就微调
自然标注思路：工程师日常使用 = 天然标注过程，知识库越用越准，高质量 Golden Dataset 自然沉淀。
优先级：先收集反馈 → 优化检索策略 → 优化 Prompt → 再考虑微调。

## Query Rewrite（多轮对话）
解决指代消解："它怎么配置" → "Wireshark 能否按 IP 地址过滤抓包数据"。
展示改写词增加用户信任感（"🔄 优化检索词: Wireshark IP 过滤"）。
