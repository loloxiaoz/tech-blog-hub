Prompt Caching 201

APICachingPerformance
Prompt Caching 进阶实践指南：从技术原理（KV Cache 跳过 prefill 计算）到成本影响（最高节省 98.75%）、延迟影响（最高降低 80%），再到提升缓存命中率的战术手册（稳定前缀、prompt_cache_key、Responses API、Flex Processing），以及 Realtime API 中的 retention_ratio 截断策略。

原文链接：https://cookbook.openai.com/examples/prompt_caching_201

Prompt Caching 基础

模型提示词通常包含重复内容——例如系统指令、工具和消息。当请求包含系统最近处理过的前缀时，OpenAI 可以将其路由到已计算过该前缀的服务器，使模型可以复用之前的工作而不是从头重新计算。

Prompt Caching 可以将首 token 延迟降低最多 80%，将输入 token 成本降低最多 90%。它自动在所有 API 请求上工作，无需额外费用。

  • 缓存命中需要精确的重复前缀匹配，自动适用于包含 1024 个或更多 token 的提示词，以 128 token 为增量发生缓存命中。
  • 整个请求前缀均可缓存：消息、图像、音频、工具定义和结构化输出模式。
  • 内存提示词缓存在所有 API 请求上自动工作。扩展提示词缓存将其增加到 24 小时。
  • 缓存仅在两个请求共享相同前缀并落在同一台机器上时有效。使用可选参数 prompt_cache_key 可改善路由。

为什么缓存很重要

核心技术原因：跳过 Prefill 计算

通过 transformer 层对输入 token 的前向传递是推理成本和延迟的主要驱动因素。Prompt Caching 专门应用于注意力层内的键值投影。KV 缓存跨所有层和头存储该前缀的键和值张量。

如果你再发出一个请求——前缀相同——处理新的、未见过的后缀时，模型会复用缓存的张量，只计算新 token 的注意力。语义基础已经建立；模型只是在现有注意力状态之上构建，而不是从头重新计算。

成本影响

缓存折扣可能很显著。以下是一些示例（完整列表见定价页面）：

模型输入（每 1M tokens）缓存输入（每 1M tokens）缓存折扣

延迟影响

减少首 token 时间（TTFT）是提高缓存率的主要动机。缓存 token 可以将延迟降低最多约 80%。对于最短的提示词（1024 token），缓存请求快 7%，但在更长的端（150k+ token），我们看到 TTFT 快 67%。输入越长，缓存对首 token 延迟的影响越大。

首先测量缓存（以便迭代）

响应包含使用字段，指示有多少提示词 token 来自缓存。所有请求将在 usage.prompt_tokens_details 中显示 cached_tokens 字段：

"usage": {
  "prompt_tokens": 2006,
  "completion_tokens": 300,
  "total_tokens": 2306,
  "prompt_tokens_details": {
    "cached_tokens": 1920
  }
}

提升缓存命中率战术手册

4.1 发送超过 1024 token 的提示词

有时，使你的提示词稍微更长可以降低总体成本。假设你有一个 900 token 的提示词——你永远不会获得缓存命中。如果你将提示词延长到 1100 token 并获得 50% 的缓存率，你将节省 33% 的 token 成本。一旦你越过缓存阈值并实现有意义的复用，这些重复 token 的边际成本就会大幅下降。

4.2 稳定前缀

这是最省力、影响最大的优化：保持提示词早期部分的稳定。将持久内容放在开头：

  • 指令
  • 工具定义
  • 模式

将易变内容（用户输入、动态值、会话特定数据）移到末尾。早期 token 的即使细微变化也会使精确前缀匹配无效并阻止缓存命中。

提示：使用 metadata

我们见过客户因在请求早期包含时间戳（用于后续查找/调试）而意外使其缓存失效。将其移到 metadata 中，这不会影响缓存！

4.3 保持工具和模式相同

工具、模式及其排序对缓存前缀有贡献——它们在开发者指令之前注入，这意味着更改它们会使缓存失效。这包括：模式键更改、工具排序更改、指令更改。

提示：在不破坏提示词缓存的情况下调整工具

利用 allowed_tools tool_choice 选项。它允许你在不更改工具数组（不破坏缓存）的情况下限制模型在请求上可以调用的工具。

# 静态"工具"保留在缓存的提示词前缀中：
tools = [get_weather_def, get_location_def, calendar_def, ...]

# 每次调用的"allowed_tools"只存在于请求元数据中，不在前缀中：
allowed_tools = {"mode":"auto", "tools":["get_weather","get_location"]}

4.4 使用 prompt_cache_key 提高路由粘性

缓存仅在两个请求共享相同前缀并落在同一台机器上时有效。请求基于提示词前 ~256 个 token 的哈希路由到推理引擎。当你提供 prompt_cache_key 时，它与该哈希结合以增加路由粘性。

一个编码客户通过开始使用 prompt_cache_key 将命中率从 60% 提高到 87%。

推理引擎每个前缀 + prompt_cache_key 组合每分钟可以处理大约 ~15 个请求。这意味着你应该选择使每个前缀 + 键组合低于 ~15 RPM 的键粒度。

4.5 使用 Responses API 而不是 Chat Completions

与 Chat Completions 相比，Responses API 的请求缓存利用率提高 40-80%。这是因为，与 Chat Completions 不同，原始思维链 token 通过 previous_response_id 在 Responses API 的轮次之间持久化。Chat Completions 没有提供持久化这些 token 的方法。

4.6 对上下文工程保持深思熟虑

当你删除、总结或压缩对话中的早期轮次时，你会破坏缓存。在这方面，上下文工程和提示词缓存本质上是相互对立的——一个优先考虑动态性，另一个优先考虑稳定性。

实用规则：利用你的 eval 来选择在成本（减少总输入 token 的截断/摘要以及缓存）和通过精心上下文工程获得的智能之间取得平衡的压缩方法和频率。

故障排除：为什么你可能看到较低的缓存

  • 工具或响应格式模式更改
  • 命中模型上下文窗口的简单截断
  • 指令或系统提示的更改
  • 推理努力度的更改
  • 缓存过期：经过太长时间，保存的前缀被丢弃
  • 添加空格、时间戳或其他动态内容
  • 将 Chat Completions 与推理模型一起使用（隐藏的思维链 token 被丢弃）

扩展提示词缓存与零数据保留

扩展提示词缓存通过在内存满时将键/值张量卸载到 GPU 本地存储来工作，显著增加可用于缓存的存储容量。

{
  "model": "gpt-5.1",
  "input": "Write me a haiku about froge...",
  "prompt_cache_retention": "24h"
}

KV 缓存只保存模型的键/值张量（注意力层产生的隐藏状态的线性投影）——这是模型内部的中间表示，意味着无论保留策略如何，原始文本/多模态输入从未被存储。

Realtime API 中的缓存

Realtime API 中的缓存与 Responses API 的工作方式相同——所有传入的音频、文本或图像都将被缓存，前缀的任何更改都会破坏缓存。然而，鉴于 Realtime API 较短的上下文窗口（目前 32k），管理截断尤为重要。

Retention Ratio

默认情况下（truncation: "auto"），服务器仅删除足够的旧消息以适应上下文窗口。这种"即时"修剪每次超过限制时都会稍微移动对话的开始，导致频繁的缓存未命中。

retention_ratio 设置通过让你控制保留多少早期上下文来改变这一点。例如，retention_ratio: 0.7 意味着在截断发生时，系统保留大约 70% 的现有对话窗口，丢弃最旧的 ~30%。

{
  "event": "session.update",
  "session": {
    "truncation": {
      "type": "retention_ratio",
      "retention_ratio": 0.7
    }
  }
}

这创建了一个更稳定的前缀，在多个轮次中存活，减少重复的缓存破坏。权衡是截断发生在更大的块中，因此你可能一次失去更多对话历史。

结论

Prompt Caching 是 OpenAI 平台上可用的最高杠杆优化之一。当你的前缀稳定且路由良好时，你可以显著降低成本和延迟，而不改变模型行为或质量。

关键思想简单但强大：稳定前缀，监控 cached_tokens，注意 ~15 RPM 限制，并深思熟虑地使用 prompt_cache_key。缓存是尽力而为且路由感知的，但经过有意工程，它可以在效率方面带来显著改善。
