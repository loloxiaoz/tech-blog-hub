Realtime Eval Guide

2026-05-15 · 约 12 分钟
Realtime Eval Voice
本指南介绍如何构建实时语音系统评测体系，从"爬行"（单轮合成音频）到"行走"（真实噪音音频）再到"运行"（多轮模型模拟）逐步提升复杂度。核心是三个构建模块：数据集（benchmark）、评分器（graders）和评测框架（eval harness）。投入 eval 体系的团队可以将产品上线速度提升 5-10 倍。
原文链接：https://cookbook.openai.com/examples/Realtime_eval_guide

引言

Eval 是将语音演示变成人们可以依赖的东西的关键。"看起来还好"和"每天都正常工作"之间的差距几乎总是 eval。

本指南展示如何通过逐步构建复杂度来评估语音系统：从简单开始（爬行），增加真实性（行走），然后测试多轮（运行）。投资 eval 的团队可以将上线速度提高 5-10 倍，因为他们可以看到失败在哪里、找出原因，并有信心地修复它。

为什么实时 Eval 很难

实时比文本更难，因为你要对一个流式交互进行评分，有两个输出：助手做了什么和听起来怎么样。一个响应可以是"正确的"，但听起来仍然很差。

实时质量的两个轴

大多数实时 eval 可以归结为两个独立的轴：

  1. 内容质量：助手是否理解了用户并做了正确的事情？正确性、有用性、工具选择、工具参数和指令跟随。
  2. 音频质量：助手听起来是否可以接受？自然性、韵律、发音、稳定性，以及在噪音和不完美捕获下的行为。

Transcript ≠ 真实情况

在 Realtime API 中，"用户说了什么"的真实情况是实际的音频信号（麦克风捕获的内容和模型听到的内容）。Transcript 不是真实情况，它是对音频的模型生成的解释，可能因转录模型错误而出错。

最佳实践：

  • 改善转录：迭代转录提示词，尝试不同模型。
  • 使用 transcript 来扩展规模：在 transcript + traces 上运行大多数自动化评分。
  • 添加音频审计循环：对 ~1-5% 的会话进行端到端抽样检查。

爬行 / 行走 / 运行策略

如果你的系统无法爬行，就无法运行。早期 eval 应该足够简单，使失败可诊断、可重复且迭代成本低。

轴1：隔离输入条件（清晰 vs 生产音频）

  • 从合成音频开始 → 测试智能：使用干净、合成的可重复音频（如TTS），在不受音频差异干扰的情况下测量模型的推理和决策。
  • 移向嘈杂的生产类音频 → 测试音频感知：一旦智能稳定，引入类似生产环境的音频：压缩、回声、远场捕获、背景噪音、犹豫/自我纠正。

轴2：隔离交互条件（单轮 vs 多轮）

  • 从单轮开始 → 测试核心能力：运行一个请求 → 一个响应，获得最干净的基础信号：正确的意图路由、正确的工具选择、有效的参数。
  • 移向多轮 → 测试鲁棒性：一旦单轮稳定，进入多轮，系统必须跨轮保持目标和约束，正确排序工具，从工具失败和用户纠正中恢复。

单轮告诉你是否能赢得战斗；多轮告诉你是否能赢得战争。

Eval 四象限

使用 2x2 图进行评估：向右 = 更真实的音频，向上 = 更真实的交互。从左下角开始，一次增加一个轴的难度：

  1. 爬行（左下）：合成音频 + 单轮
  2. 行走（向右移动）：真实嘈杂音频 + 单轮
  3. 运行（向上移动）：合成音频 + 多轮模拟

右上角（真实音频 + 完整多轮流程）是手动 eval：像用户在生产中那样进行端到端会话。

三个构建模块

数据集：构建 Benchmark

从"黄金"种子集开始（10-50个）

覆盖你不能失败的流程：核心意图、必须工作的工具调用、升级和拒绝行为。快速生成，然后让人工审查真实性和差距。目标是开始，而不是完美。

为迭代而构建，而不是为了数量

Eval 数据集存在是为了驱动迭代，而不是为了看起来很大。循环就是产品：运行 eval → 将失败定位到特定行为 → 改变一件事 → 重新运行 → 确认修复有所改善且没有回归。

覆盖率也必须是平衡的。对于每种行为，同时包含正例（系统应该做X）和反例（系统不应该做X）。没有反例，你就会奖励捷径。

从生产失败中扩展

操作模式是一个循环：生产扩展 benchmark。出现新的失败，你重现它，标记它，并添加它。三个集合：

  • 回归套件：你已经修复的难案例。每次提示词、模型和工具变更都运行。
  • 滚动发现集：生产中的新鲜失败和险情。这是你学习缺失内容和下一步优先级的地方。
  • 保留集：偶尔运行以检测 benchmark 过拟合的离线测试子集。

评分器

评分器是你的测量工具，将混乱的实时语音会话转化为你可以信任的信号。

手动审查（最高杠杆）

手动审查 = 听真实音频 + 端到端阅读完整 traces。这是建立产品直觉和发现用户立即注意到的失败的最快方法。自动化 eval 告诉你可以测量什么。手动审查告诉你应该测量什么。

自动化评分器

使用分层评分器堆栈：

  1. 确定性评分器：用于任何客观且机器可检查的内容。快速、便宜、稳定。适合工具调用、JSON有效性、字符串和模式检查。
  2. LLM 评分器：帮助测量重要但不适合确定性规则的内容：正确性、指令跟随、澄清是否适当、完整性和有用性。
  3. 音频评分器：因为用户体验的是声音，而不是 transcript。从简单、可测量的检查开始（静默、重叠、中断处理），随时间添加更丰富的评估标准。

评测框架

实时 eval 的可信度取决于运行它的框架。好的框架只有一个任务：使运行可比较。

单轮重播框架（"爬行"框架）

从这里开始。单轮重播提供最快、最干净的信号，因为你可以保持几乎所有内容固定。在实践中，通常最好先关闭语音活动检测（VAD），这样可以消除一个主要的差异来源。

保存音频重播（"行走"框架）

当你从合成TTS移动到真实录音时，框架只在一个重要方面改变：你从保存的真实音频流式传输音频缓冲区。预处理必须与生产匹配：相同的重采样、归一化、通道处理。

模型模拟多轮（"运行"框架）

模型模拟多轮使用用户模拟器生成完整对话的下一个用户轮次。最佳实践：

  • 固定和版本化模拟器提示词：将其视为代码。
  • 约束随机性：固定温度和采样设置。
  • 确定性地模拟工具：定义预期的工具输出模拟，在助手调用工具时返回这些确切的输出。
  • 记录完整轨迹：存储每个生成的用户文本轮次以及最终的音频字节。

案例研究：客户支持语音机器人

产品目标：通过工具快速、安全地解决常见支持请求。机器人必须收集正确的详细信息，调用正确的后端操作，并遵守政策。

爬行、行走、运行计划：

  • 爬行：合成 + 单轮。专注于路由和策略。
  • 行走：真实 + 单轮。在真实捕获条件下测试理解。这是订单号、名称和地址在嘈杂音频下崩溃的地方。
  • 运行：合成 + 多轮模拟。用 gpt-realtime 模拟完整工作流和工具模拟。
  • 手动审查：针对暂存系统运行内部呼叫会话，捕获评分器遗漏的 UX 失败。
