# 构建高效的智能体

在过去一年里，我们与数十个团队合作构建大语言模型（LLM）智能体。最成功的实现并没有使用复杂框架或专用库，而是用**简单、可组合的模式**搭建起来。

## 什么是智能体

"智能体"有多种定义。我们在 Anthropic 把这些变体统称为**智能体系统（agentic systems）**，但在架构上区分两类：

- **工作流（Workflows）**：通过预定义代码路径来编排 LLM 和工具的系统。
- **智能体（Agents）**：由 LLM 动态决定自身流程与工具调用、自主掌控如何完成任务的系统。

> 经验法则：先找最简单的方案，只有在确有必要时才增加复杂度。

## 何时该用智能体

构建 LLM 应用时，我们建议尽量用最简单的方案，只在需要时增加复杂度。智能体系统通常用**更高的延迟和成本**换取更好的任务表现，你需要判断这个权衡是否划算。

1. 对定义清晰的任务，工作流提供可预测性与一致性；
2. 当需要规模化的灵活性与模型自主决策时，智能体是更好的选择；
3. 对许多应用而言，优化单次 LLM 调用 + 检索 + 上下文示例就已经足够。

## 何时使用框架

许多框架能简化智能体的实现，例如调用 LLM、定义与解析工具、串联调用等底层工作。但它们往往引入额外抽象层，**掩盖了底层的 prompt 与响应**，更难调试。

```python
# 直接用 API 往往几行就够，便于理解与调试
resp = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    messages=[{"role": "user", "content": prompt}],
)
```

我们建议开发者**先直接使用 LLM API**；若使用框架，务必理解其底层代码。

## 核心构建块对比

| 模式 | 适用场景 | 复杂度 |
|------|---------|--------|
| 单次调用 + 检索 | 大多数任务 | 低 |
| 提示链（Prompt Chaining） | 可拆解为固定步骤 | 中 |
| 路由（Routing） | 输入需分类处理 | 中 |
| 编排器-工作者 | 子任务数量不定 | 高 |
| 自主智能体 | 开放式、需多轮决策 | 高 |

## 要点总结

复杂度不是目标。从简单方案起步，用可组合的模式逐步加码，并始终保持对底层 prompt 的可见性，是构建可靠智能体的关键。
